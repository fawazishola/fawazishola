The moment everything starts to make sense.

It was 2 AM on a Tuesday when I finally committed to the most unhinged decision of my technical life: formatting my only computer to build an operating system from literal source code.

Not installing Linux. Not dual-booting. Building it. Every package. Every library. Every binary ‚Äî compiled from scratch on my machine, optimized for my exact CPU.

The project is called Axiom. It's a custom Linux workstation built entirely from source, optimized for mathematical and scientific computing. And yes, I built it following the legendary Linux From Scratch book.

"But you can just install Ubuntu or Arch. Why would anyone do this?"

That's exactly what I thought too. Until I understood the difference between running a computer and understanding a computer.

The Problem With "Normal" Linux

Here's a dirty secret about standard Linux distributions: they're built for everyone. And when you build for everyone, you optimize for no one.

Standard Distro Axiom (Math-Native)
Generic binaries (-march=x86-64) CPU-optimized (-march=native)
Bloated desktop (GNOME/KDE) Minimal tiling WM (i3)
Generic BLAS/LAPACK OpenBLAS tuned for exact CPU
Desktop animations eating cycles Every cycle goes to computation

When you install NumPy on Ubuntu, it links against a generic BLAS library compiled to run on any x86 processor from the last 20 years. It can't use your AVX-512 instructions. It can't use your specific cache sizes. It's leaving performance on the table.

But when you compile OpenBLAS from source with -march=native? Your CPU's vector instructions light up like a Christmas tree.

What I Actually Built

Let me walk you through the madness. Axiom isn't just "a computer that boots." It's a 10-phase stack designed for PhD-grade computational work:

Phase 1: The Vision

Before writing a single line of code, I asked myself: What would a mathematician's dream workstation look like?

  • No bloat. No games, no office suites, no web browser, no audio/video nonsense.
  • Every byte justified. If it's installed, it serves the mission.
  • Native performance. CPU-specific optimizations everywhere.

Phase 2: The "Infinite Precision" Triad

These three libraries form the bedrock of computational mathematics:

  • GMP ‚Äî Arbitrary precision integers and rationals. Calculate œÄ to a million digits.
  • MPFR ‚Äî Reliable floating-point with guaranteed correct rounding.
  • MPC ‚Äî Complex arbitrary precision, built on GMP + MPFR.

Build order matters. GMP ‚Üí MPFR ‚Üí MPC. Get them wrong and mysterious segfaults await you at 3 AM.

Phase 3: Heavy Metal Linear Algebra

This is the engine behind NumPy, Octave, R, TensorFlow ‚Äî if this layer is slow, everything is slow.

export CFLAGS="-O3 -march=native -pipe"
export CXXFLAGS="${CFLAGS}"
export MAKEFLAGS="-j$(nproc)"

Those three lines are the secret weapon. -O3 for aggressive optimization. -march=native to use your CPU's exact instruction set. -j$(nproc) to use all your cores.

‚ö†Ô∏è Critical: OpenBLAS

Do NOT install the generic libblas package. Build OpenBLAS from source with your exact CPU target. This single decision can mean 15‚Äì20% faster matrix operations.

Phase 4: The "Oracle" ‚Äî Embedded AI

Here's where Axiom gets weird. I'm embedding a local LLM ‚Äî not through Python or PyTorch, but by compiling llama.cpp from source with native optimizations.

The model runs at maximum speed using my CPU's vector instructions. No Docker. No pip install. Just C++ and math.

cmake -B build \
    -DGGML_NATIVE=ON \
    -DBUILD_SHARED_LIBS=ON \
    -DCMAKE_BUILD_TYPE=Release

cmake --build build -j$(nproc)

A 7B parameter model in 5GB of RAM, running 50+ tokens per second, verified by a custom language I built called Tenet that acts as a game-theory referee for LLM outputs.

But that's a story for another article.

The Journey: What They Don't Tell You

Linux From Scratch is a 400+ page book. It takes 40‚Äì60 hours for a first-timer. And it will break you.

The First Kernel Panic

At hour 32, I finally had a bootable system. The GRUB menu appeared. The kernel started loading. Then: 

Kernel panic - not syncing: VFS: Unable to mount root fs

I stared at that message for 20 minutes. Then I opened Chapter 10 again. Then I realized I'd misconfigured my filesystem options in the kernel.

Four hours later, after recompiling the entire kernel, I watched the login prompt appear.

I've never felt that kind of satisfaction from apt install.

The "Quick" Package That Took 3 Days

GCC (the GNU Compiler Collection) is a monster. 4.5 hours to compile on my machine. And when it fails at hour 4, you get to start over.

The book tracks build time in "SBUs" ‚Äî Standard Build Units, normalized to how long your first package takes. GCC is 155 SBUs. That's not a typo.

At some point, you stop seeing packages. You start seeing dependencies. You understand that libc underlies everything. That coreutils is just GNU's implementation of POSIX. That nothing is magic.

What I Actually Learned

Here's the uncomfortable truth: you don't need to do this. Pre-built binaries work fine. Ubuntu is genuinely excellent. Most developers never need to understand what ./configure actually does.

But if you're the kind of person who needs to know, there's no substitute for building the thing yourself.

1. Nothing Is Magic

When you've compiled glibc from source, you understand what system calls actually are. When you've built the kernel, you understand what drivers do. When you've linked OpenBLAS, you understand why matrix multiplication can be fast or slow.

2. Dependency Hell Is Real (And Navigable)

Build order matters. Build flags propagate. One misconfigured library can cascade through your entire stack. But once you understand this, you can debug it ‚Äî instead of just reinstalling and hoping.

3. Performance Is a Choice

Every cycle your desktop burns on animations is a cycle not computing your eigenvalues. Axiom has no compositor, no themes, no rounded corners. Just a tiling window manager and the raw power of -march=native.
